# Auto-Derive Seeded Facts Plan
*Save to: `projects/kwaai-knowledge/plans/AutoDeriveSeededFacts-plan.md`*

## Context

The broad mission is a fully automated RAG pipeline running on consumer hardware with open-weights local models — no frontier-model knowledge injection, no manual curation. Currently the D6 knowledge base depends on a significant body of manually entered facts: 74 family relations, 18 entity descriptions, 1 manual override, and several hand-annotated schema files — all authored by Claude/NotebookLM or human curation. This plan inventories every manually entered fact, maps each to an automated derivation technique, identifies which techniques exist vs. are missing, and proposes an implementation sequence to close the gap.

## Current Automated Techniques vs. Gaps

### What Already Works Automatically

| Technique | File | Capability |
|-----------|------|-----------|
| Entity extraction (names, types, fields) | `graph.rs` | Extracts Person/Place/Org/Concept with structured fields from chunks |
| Entity enrichment (descriptions) | `enrich.rs` | Generates 3–5 sentence descriptions from evidence chunks; distinctive-alias filtering; corpus fallback; hallucination warnings |
| Gender extraction | `enrich.rs` | Pronouns + titles → Male/Female/null |
| Entity deduplication (Tiers 1–3) | `graph.rs` | Exact → similarity → abbreviation matching |
| GLiNER NER pre-screening | `ner.rs` | Person span detection; feeds as hints to extraction prompt |
| Coreference candidates | `graph.rs:coref_candidates_for_chunk()` | Identifies pronoun candidates per chunk (partially implemented) |
| Document metadata extraction | `doc_schema.rs` | ISBN, publisher, copyright year, author, section types |
| Timeline event extraction | `sequence.rs:extract_temporal_events()` | Dated events + two-entity interactions (implemented but not wired into build) |
| Entity confidence scoring | `scorer.rs` | 3-pillar: type + summary + relation scores |
| Entity-centric (EC) extraction | `graph.rs` | One LLM call per unique name with aggregated evidence |

### Phase 4 — Relation Extraction with Lexical Pre-Filter *(2–3 days)*

**STATUS: ✅ COMPLETED (2026-06-26)**  
`lexical_relation_trigger()` added to `graph.rs`. `extract_from_text()` computes
`effective_no_relations = no_relations || !lexical_relation_trigger(text)` — relation
extraction LLM call only fires on chunks with kinship/membership keywords.
Cross-chunk support filter added to `retriever.rs:build_relations_suffix()` — requires
≥2-chunk evidence for LLM-extracted relations; seeded relations always pass.
D6 still uses `--no-relations` (trigger only activates when relations are enabled).

**Goal**: Re-enable relation extraction for family/political relations with high enough precision to reduce YAML seed dependency.

**Problem**: 8b models produce 0–17% precision on unconstrained relation extraction. The fix is to only run relation extraction on chunks containing explicit kinship or membership trigger words.

**Changes**:
1. In `graph.rs` relation extraction path: add `lexical_trigger_filter(chunk_text, relation_types)` — only attempt relation extraction on a chunk if it contains trigger words (e.g., "son of", "wife", "married", "sibling", "founded", "member of", "affiliated with")
2. Add cross-chunk support counter: a relation must appear in ≥2 chunks before being stored (reduces hallucinated one-off relations)
3. Add family relation guards (R1: cannot be both parent_of and sibling_of same entity; R2: if same parent but different co-parent → half_sibling_of not sibling_of; R3: high-risk surnames like Gool/Rassool require entity_id match not just name match before merging relations)
4. Add `--enable-relations` flag to `rag graph build` (currently hardcoded `no_relations=true` for D6 via config); leave D6 default as disabled until precision is validated
5. Validate against family tree YAML: `kwaainet rag graph validate-relations --kb D6 --ground-truth tests/kwaai-knowledge/d6_family_tree.yaml`

**Files**: `core/crates/kwaai-rag/src/graph.rs`, `core/crates/kwaai-cli/src/rag_cmd.rs`, `core/crates/kwaai-cli/src/rag_cmd.rs`

### Phase 5 — Coreference Resolution *(2–3 days)*

**STATUS: ✅ ALREADY IMPLEMENTED (discovered 2026-06-26)**  
`kwaainet rag graph coref --kb D6 [--no-llm] [--commit]` already exists with full
3-tier approach: alias match → gender/pronoun (rule-based) → LLM disambiguation.
Post-rebuild step: `rag graph coref --kb D6 --no-llm --commit` to add chunk-entity links.

**Goal**: Map narrator pronouns ("I", "my", "he", "she") to canonical entities so that pronoun-rich passages contribute to entity-chunk links and relation extraction.

**Why this matters for auto-derivation**: The memoir is ~70% pronoun-heavy narrative. "My grandfather's house" should link to JMH Gool. Without coreference, relation extraction sees "my brother Fazil" and picks wrong endpoints.

**Changes**:
1. Tier 0: `coref_candidates_for_chunk()` already exists in `graph.rs:1434` — extend to return (pronoun, candidate_entity_id, confidence) triples
2. Tier 1 (rule-based): kinship phrases ("my grandfather", "his wife", "their son") resolved by walking the known graph from narrator entity; store as chunk-entity links
3. Tier 2 (LLM): for unresolved pronouns, call inference with a focused disambiguation prompt: "In this passage, 'he' refers to which of: [candidate list]?"
4. New CLI: `kwaainet rag graph coref --kb D6 --narrator "Yousuf Rassool" [--commit]`; dry-run shows proposed resolutions, `--commit` writes them

**Files**: `core/crates/kwaai-rag/src/graph.rs` (coref_candidates_for_chunk extension), new Tier 1/2 resolution functions, `core/crates/kwaai-cli/src/rag_cmd.rs`

**STATUS: ✅ ALREADY IMPLEMENTED (discovered 2026-06-26)**  
`summary.rs`, `SUMMARY_NODES_TABLE`, `kwaainet rag summarize --kb D6 --window-size 10`,
and retriever Round 2.5 (summary node cosine search) already exist.
Post-rebuild step: `rag summarize --kb D6 --inference-url "p2p://..." --window-size 10`

**Goal**: Aggregate multi-paragraph narratives (forced removals, cricket involvement, political movement history) into chunk-level summaries that survive retrieval.

**Changes**:
1. Create `core/crates/kwaai-rag/src/summary.rs` with `SummaryNode` struct {id, chunk_ids, summary_text, embedding, section_path}
2. Add `SUMMARY_NODES_TABLE` to the redb meta-store schema
3. `kwaainet rag summarize --kb D6 --window 5` — groups chunks into 5-chunk windows, calls LLM to generate summary preserving all named entities/dates/places
4. Insert summaries as "Round 2.5" in retrieval: after entity expansion, before LLM reformulation, fetch summary nodes covering the retrieved chunk range
5. Summary nodes are NOT added to BM25 index (too lossy); used only for context expansion

**Files**: new `core/crates/kwaai-rag/src/summary.rs`, `core/crates/kwaai-rag/src/meta_store.rs` (new table), `core/crates/kwaai-rag/src/retriever.rs` (Round 2.5 insertion), `core/crates/kwaai-cli/src/rag_cmd.rs`

## Gap Analysis: What Cannot Be Automated Yet

| Gap | Why It's Hard | Recommendation |
|-----|--------------|---------------|
| **Cause-effect chains** (Act → removal) | `causes` relation type missing; event-event temporal links not in schema | Add to entity-type schema plan as `results_in` relation; wire timeline events as graph nodes |
| **Document narrator auto-detection** | "I" in intro could be editor, translator, or ghost-writer; requires document structure analysis | Auto-detect when a `narrator_note:` field conflicts with cover-page author; fallback to requesting annotation |
| **Genealogical completeness** | Two-wife branching, half-siblings, foster children are correct ONLY with complete family tree; sparse extraction will miss branch points | Keep YAML seed as ground-truth; use Phase 4 relation extraction to VERIFY against seed, not replace it |
| **External historical facts** | NEUM 1943 founding date, Abdurahman's Glasgow degree dates — not in memoir text | Annotate `evidence_source: external` in YAML; do not attempt to extract from document |
| **Quote attribution / document provenance** | "Gandhi says X" vs. "Indian Opinion reports X" vs. "memoir recalls X" requires source-level tracking through the full pipeline | Requires new `provenance` field on RelationRecord; complex multi-document architecture; no plan exists |
| **Interactive dedup review** | OCR typos (Gooli, J_ M_ H_) require human confirmation before canonical rename | Add `kwaainet rag graph rename-entity --kb D6 --from X --to Y --pin` command; `pinned: true` flag blocks dedup |

## Priority & Sequence

```
Phase 1 (AutoDescriptions)  →  Phase 2 (Timeline wiring)  →  Phase 3 (Entity-type schema)
                                                                     ↓
Phase 5 (Coreference)  ←──────────────────────────────────  Phase 4 (Relation extraction)
                                                                     ↓
                                                             Phase 6 (Hierarchical summaries)
```

Phases 1–3 are independent and can run in any order. Phase 4 benefits from Phase 3 (proper entity types guide relation filtering). Phase 5 unlocks better Phase 4 results (narrator visible in chunks). Phase 6 can start in parallel with Phase 5.
