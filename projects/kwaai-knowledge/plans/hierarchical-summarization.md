# Sub-project B: Hierarchical Summarization (HiRAG/TreeRAG-lite)

## Context

RAG over long documents (200-page memoirs, books) has a fundamental gap between
chunk-level retrieval and document-level understanding. A single 100-word chunk may
be the right answer, but the query needs "who was JMH Gool?" resolved at the level of
an entire chapter, not one paragraph. HiRAG/TreeRAG-style systems build a second index
of multi-chunk summaries and retrieve at that level first, then expand to base chunks.

This sub-project adds a two-level summary layer that:
1. Generates compact summaries of N-chunk windows at ingest time (or on-demand)
2. Stores summaries with embeddings in a new `SUMMARY_NODES_TABLE` in MetaStore
3. Uses summary embeddings as a first-pass retrieval signal in the iterative pipeline

The summary infrastructure reuses the existing dream cycle's LLM call pattern,
concurrent worker model, and evidence trimming — no new HTTP/LLM client needed.

## Implementation — 3 phases

### Phase 1 — Summary storage (½ day)

**New struct** in a new `core/crates/kwaai-rag/src/summary.rs`:
```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct SummaryNode {
    /// Deterministic ID: sha256("summary::" + doc_name + "::" + first_chunk_id)[..8]
    pub id: i64,
    /// Level 0 = window summary (N base chunks); Level 1 = section summary (M window summaries)
    pub level: u8,
    /// Compact prose summary generated by LLM
    pub text: String,
    /// Embedding of text with "search_document: " prefix
    pub embedding: Vec<f32>,
    /// IDs of the direct children (base chunk IDs at level 0; SummaryNode IDs at level 1+)
    pub child_ids: Vec<i64>,
    /// Section name inherited from the first child chunk, if available
    pub section_name: Option<String>,
    pub doc_name: String,
    pub created_at: String,
}
```

**New table in `meta_store.rs`**:
```rust
const SUMMARY_NODES_TABLE: TableDefinition<&[u8], &[u8]> =
    TableDefinition::new("summary_nodes");
// key = tenant_uuid[16] + summary_node_id_i64_le[8] → SummaryNode JSON
```

**New `MetaStore` methods**:
```rust

pub fn get_summary_nodes(&self, ids: &[i64], tenant_id: Uuid) -> Result<Vec<Option<SummaryNode>>>
```

### Phase 2 — Summary generation (1 day)

**New module `summary.rs`**: `generate_summaries(chunks, meta, embed, cfg)` runs the
two-level bottom-up pass.

**Level-0 (window summaries)**:
1. Group base chunks into windows of `window_size` (default 5, configurable).
2. For each window, concatenate texts → `trim_evidence()` → call LLM with prompt:
   ```
   Summarize the following {N} passages in 2–3 sentences, capturing the key people,
   events, and facts. Be specific — include names, dates, and places.
   Text:
   {chunk_texts}
   ```
3. Embed summary text via `embed_batch()`.
4. Store as `SummaryNode { level: 0, child_ids: [chunk_ids], ... }`.

**Level-1 (section summaries, if DocSchema has sections)**:
1. Group level-0 nodes by `section_name`.
2. Summarize the concatenated level-0 summaries per section.
3. Store as `SummaryNode { level: 1, child_ids: [level0_node_ids], ... }`.

**Concurrency**: Reuse the `DreamConfig.workers` + `tokio::sync::Semaphore` pattern
from `dream.rs` — same concurrent fan-out, same channel-based drain.

**LLM call**: Reuse `call_llm(prompt, url, model)` from `dream_tasks.rs:220`.

**CLI command** — `kwaainet rag summarize`:
```
kwaainet rag summarize --kb D6 \
  --window 5 \
  --levels 2 \
  --workers 6 \
  --inference-urls http://localhost:11434 \
  --model llama3.1:8b
```
Runs after ingest. If summaries already exist for a document, skips unless `--force`.

**GraphAction integration in `rag_cmd.rs`**: Add `RagAction::Summarize` dispatch.

### Phase 3 — Hierarchical retrieval (1 day)

**Retrieval augmentation in `iterative.rs` Round 2.5**:

After Round 2 (graph entity gap-fill), if coverage is still below threshold, try
summary expansion:

```rust
// Round 2.5: summary-node expansion
let summary_ids = search_summaries(&query_emb, meta, tenant_id, 5);  // top-5 summaries
for sid in summary_ids {
    if let Some(node) = meta.get_summary_node(sid) {
        let child_chunks = meta.get_chunks(&node.child_ids);
        // add new child chunks to pool at score 0.40
        pool.extend(child_chunks.into_iter()
            .filter(|c| !seen.contains(&chunk_key(c)))
            .map(|c| RetrievedChunk { score: 0.40, ... }));
    }
}
```

`search_summaries()` — brute-force cosine similarity over all `SummaryNode.embedding`
values for the tenant. Fast enough at D6 scale (< 1000 summary nodes).

**Score for summary-expanded chunks**: 0.40 (between graph gap-fill 0.45 and LLM
reformulation) — they are informative but not direct matches.

**New `RetrieveConfig` field**:
```rust
pub use_summary_expansion: bool,  // default true if summary nodes exist
```

**`retrieve_hybrid()` fast path**: When `top_k` is small and all coverage terms are
found in Round 1, skip summary expansion entirely — it's only for gap-filling.

## Sequencing / dependency note

**Implement after Sub-project A** (`structure-aware-ingestion.md`).

Sub-project A Phase 1 (SectionType enum on chunks) directly feeds Phase 2 of this plan:
- With section boundaries, level-1 summary nodes align to chapters/sections rather than
  arbitrary windows of N window-summaries
- `SectionType.Chapter` boundaries → natural split points for level-0 windows
- `SectionType.FrontMatter`, `.TableOfContents`, `.Index` → skip during summarization

Without Sub-project A, fall back to window-based grouping (window=5 consecutive chunks).
That's a valid first iteration but section-aligned summaries will be sharper.

Within this plan: Phase 1 (storage) → Phase 2 (generation) → Phase 3 (retrieval).
Phases 2 and 3 are independently testable — validate summary quality before wiring
into retrieval.
